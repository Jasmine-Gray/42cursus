## 1. 全体のアーキテクチャ：データの橋渡し

役割を明確に分けるために2つの主要な構造体を使用。

* **`t_game` (パーサー用)**: ファイルから読み取った「生のデータ」を一時的に保管する箱。
* **`t_data` (レンダリング用)**: 実際に画面を描画するために必要な「計算用データ」を保持する箱。

`main.c` で `setup_data` を実装したことで、パーサーが苦労して集めたマップや色情報が、描画エンジンに正しく引き継がれるように。これができていないと、計算機（レンダリング）に数字が入っていない状態になり、クラッシュが起こりうる。

---

## 2. 空間の解釈：グリッドと座標系

コンピュータにとって、マップはただの「文字の羅列」。それを `parse_map` で **2次元配列 (`char **grid`)** に変換。

* **世界座標**: マップの `(x, y)` は、数学的な座標平面として扱われる。
* **中心への配置 (`+ 0.5`)**: プレイヤーの座標を `(5.5, 10.5)` のように設定することで、1x1マスの正方形のちょうど真ん中に立たせる。これにより、壁に埋まるリスクを減らし、計算を安定させる。

---

## 3. 描画の核心：レイキャスティング・アルゴリズム

`raycast.c` で行っているのは、**「光線投射（Raycasting）」** という手法。

### DDA (Digital Differential Analyzer)

これが今回最も重要なアルゴリズム。
3Dゲームにおいて「壁に当たったか」を判定するために、光線を 0.001 ずつ進めてチェックするのは非常に非効率。そこで **DDA** を使う。

* **仕組み**: 光線が次に「マスの境界線（縦線または横線）」をまたぐ瞬間だけを計算して、ワープするように進む。
* **メリット**: 計算回数が劇的に減り、1990年代の非力なコンピュータでもリアルタイムで3Dを表示できた。

---

## 4. 投影の数学：魚眼レンズの補正

光線が壁に当たったら、その距離を測定。しかし、単にプレイヤーからの直線距離を使うと、画面の両端が歪む **「魚眼レンズ効果」** が発生してしまう。

* **解決策**: プレイヤーの向きに対して **垂直な距離（Perpendicular distance）** を計算。これが `get_perp_dist` の役割。
* **3D化の魔法**: 壁の高さ  は、距離  に反比例。このシンプルな計算だけで、遠くの壁は小さく、近くの壁は大きく描画され、奥行きが生まれる。

---

## 5. グラフィックの実装：MiniLibX

描画には MiniLibX を使用していますが、標準の `mlx_pixel_put` ではなく、**イメージバッファへの直接書き込み**（`my_pixel_put`）を実装。

* **なぜ？**: `mlx_pixel_put` は1ピクセルごとにOSに描画を依頼するため非常に遅い。
* **高速化**: メモリ上に「1フレーム分の画像の設計図」を一気に作り、最後にドカンと1回だけ画面に転送することで、滑らかな描写（60FPS以上）を可能に。

## 1. DDA (Digital Differential Analyzer) アルゴリズムの詳細

DDAは、光線がマップ上の壁（グリッド）に衝突する最短距離を、浮動小数点の誤差を抑えながら計算する手法。

### 1.1 核心となる準備計算

光線1ステップ分進むときの「増分距離」を求める。

* `deltaDistX = sqrt(1 + (rayDirY * rayDirY) / (rayDirX * rayDirX))`
* 簡略化：`deltaDistX = fabs(1 / rayDirX)` （比率が同じなため、こちらが一般的） 





### 1.2 SideDist（初期距離）の計算

現在のプレイヤー位置（`posX`, `posY`）から、**最初のグリッド線**までの距離を計算。

* **x方向が負（左向き）の場合**:
`stepX = -1`, `sideDistX = (posX - mapX) * deltaDistX`
* **x方向が正（右向き）の場合**:
`stepX = 1`, `sideDistX = (mapX + 1.0 - posX) * deltaDistX`
※ `y` 方向も同様に計算。

### 1.3 DDA実行ループ

`sideDistX` と `sideDistY` を比較し、常に「近い方」へ1ステップ進みます。

```c
while (hit == 0)
{
    if (sideDistX < sideDistY)
    {
        sideDistX += deltaDistX;
        mapX += stepX;
        side = 0; // 垂直壁(東西)にヒット
    }
    else
    {
        sideDistY += deltaDistY;
        mapY += stepY;
        side = 1; // 水平壁(南北)にヒット
    }
    if (map[mapY][mapX] == '1') hit = 1; // 壁に衝突
}

```

### 1.4 垂直距離の計算（魚眼補正）

プレイヤーから衝突点までの直接距離ではなく、**カメラ平面に対する投影距離**を算出します。これにより、壁が歪むのを防ぎます。

* `if (side == 0) perpWallDist = (sideDistX - deltaDistX);`
* `else perpWallDist = (sideDistY - deltaDistY);`

## 2. 実装のステップ・バイ・ステップ

### STEP 1: 設定ファイルのパース (`.cub`)

* 
**テクスチャ**: `NO`, `SO`, `WE`, `EA` の識別子でパスを取得 。


* 
**色**: `F` (床), `C` (天井) を `R,G,B` (0-255) 形式で取得 。


* 
**マップ**: 最後に配置されている必要がある 。`0, 1, N, S, E, W` のみで構成され 、壁で囲まれているかチェックする 。



### STEP 2: レンダリングループ

1. 画面の各列（`x = 0` から `width`）について光線を計算。
2. DDAを実行して `perpWallDist` を算出。
3. 壁の高さ（`lineHeight = h / perpWallDist`）を計算し、描画範囲を特定。
4. テクスチャの `x` 座標（`texX`）を決定。壁のどの部分に当たったか（`wallX`）から算出。

### STEP 3: 入力制御

* 
`W, A, S, D`: プレイヤーの座標更新 。


* 
`Left, Right Arrows`: プレイヤーの方向ベクトル（`dir`）とカメラ平面（`plane`）を回転行列で計算 。


* 
`ESC` or `Red Cross`: クリーンに終了 。

## 3. Bonus機能の深掘り実装

ボーナスは、Mandatory（必須パート）が完璧な場合のみ評価されます 。

3.1 壁の衝突判定 (Wall Collisions) 

* **ロジック**: 移動後の新しい座標 `newX = posX + dirX * moveSpeed` が `map[posY][newX]` で壁でないか確認します。
* **Tip**: 滑らかさを出すため、`x` と `y` の判定を別々に行うと、壁に沿って滑るような移動が可能になります。

3.2 ミニマップ (Minimap) 

* **実装**: 画面の隅（例：左上）に、マップ配列を小さなブロックとして描画。
* **工夫**: プレイヤーの現在位置を点として表示し、向いている方向に短い線（視界ベクトル）を描くと分かりやすくなります。

3.3 扉 (Doors) 

* 
**マップ拡張**: マップファイルで `'D'` などの新しい記号を使用 。


* **DDAの修正**: `map[mapY][mapX] == 'D'` のとき、その座標の「開き具合（0.0〜1.0）」の状態変数を参照。
* **アクション**: `Space` キーなどで、一定範囲内の扉の状態変数をタイマーで増減させます。

3.4 マウス操作 (Mouse Rotation) 

* **実装**: `mlx_hook` で `MotionNotify` をキャッチ。
* **ロジック**: 前回のマウス `x` 座標との差分を回転速度に変換。画面外に出ないよう、`mlx_mouse_move` で常に中心に戻す処理を加えると操作性が向上します。

3.5 アニメーションスプライト 

* **データ構造**: スプライトの位置（`x, y`）とテクスチャ情報を格納。
* **レンダリング**: 壁を描画した後、**遠い順にソート**して描画（画家アルゴリズム）。
* **アニメーション**: フレームごとにテクスチャを切り替えるカウンタを `mlx_loop_hook` 内で更新します。

# cub3D 周辺アルゴリズム・数学メモ

## 1. Flood Fill アルゴリズム (マップバリデーション)

マップが壁（`1`）で完全に囲まれているかを判定するために使用します 。

* **用途**: パーサー段階で、プレイヤーが歩けるエリア（`0` や開始位置）から「外の世界（マップ外）」へ漏れ出さないかを確認します。
* **ロジック**:
1. プレイヤーの開始位置から上下左右に再帰的（またはスタックを使用）に探索。
2. 探索中に「マップの端」や「空白」に到達してしまったら、そのマップは閉じていない（不正）と判断します。
3. 探索済みの箇所を一時的に別の文字（例: `.`）に置き換えることで、無限ループを防ぎます。



## 2. ベクトルの回転行列 (視点移動)

プレイヤーの方向（`dirX/Y`）とカメラ平面（`planeX/Y`）を左右に回転させるために使用します 。

* **用途**: 左右の矢印キーを押した際の視点変更。
* **計算式**: 反時計回りに $\theta$ 回転させる場合、新しいベクトル $(x', y')$ は以下のように計算されます。

$$x' = x \cdot \cos(\theta) - y \cdot \sin(\theta)$$


$$y' = x \cdot \sin(\theta) + y \cdot \cos(\theta)$$


* **実装のコツ**: 回転行列を適用する際、`x` と `y` を同時に更新する必要があります。先に `x` を更新してしまうと、その後の `y` の計算に「新しい `x`」が使われてしまい、ベクトルが歪んでしまいます。

## 3. 画家アルゴリズム (Painter's Algorithm) ※Bonus

スプライト（敵やオブジェクト）を正しく描画するために使用します 。

* **用途**: 複数のスプライトがある場合、遠くにあるものから順番に描画することで、重なりを正しく表現します。
* **手順**:
1. プレイヤーからの距離を各スプライトごとに計算。
2. 距離が遠い順にソート（`sort` アルゴリズムが必要）。
3. 遠い順に描画を重ねていきます。



## 4. テクスチャ・マッピング (Texture Mapping)

壁のどの部分に、画像のどのピクセルを貼るかを決定するロジックです 。

* **用途**: 単なる色の壁ではなく、画像を表示させる。
* **ステップ**:
1. **wallX**: 壁に当たった正確な小数点の位置を算出。
2. **texX**: `wallX` をテクスチャの幅（例: 64px）に合わせてスケーリング。
3. **step**: 壁の高さ（`lineHeight`）に対して、テクスチャのピクセルをいくつ飛ばし（または重複）で描画するかを算出。



## 5. 線形補間 (Linear Interpolation) ※Bonus

影（Shadowing）やライティングの効果を出すために使われることがあります 。

* **用途**: 距離が遠くなるほど壁の色を暗くする。
* **ロジック**: `perpWallDist`（壁までの距離）に応じて、元のテクスチャの色を黒（`0x000000`）に近づけていきます。
